# FDR 0001: Claude Hooks Actions Config

## Status

**exploring** — problem statement and rough schema captured. No
implementation. Concrete v1 work (`lux fmt-all` + Stop hook) explicitly
defers this design.

## Problem

Lux's Claude Code package generates a fixed set of hooks at `generate-plugin`
time. Today there are two:

1. `PreToolUse` — deny built-in tools when an MCP equivalent exists
   (generated by `go-mcp`)
2. `PostToolUse` `Edit|Write` → `lux fmt <file>`
   (generated by `internal/hooks/generate.go`)

This shape has several known limitations:

1. **No user control over which hooks are active.** A user who finds the
   `PostToolUse` formatter disruptive (see edit-loop pathology in the
   v1 design doc) has no toggle short of editing the package's generated
   `hooks.json` by hand.

2. **No per-filetype customization.** Markdown formatting in particular has
   different ergonomics than Go or Rust formatting — the agent's edits get
   reflowed in ways that invalidate downstream `Edit` calls. The user
   should be able to opt markdown out (or any specific filetype).

3. **No way to bind new lux subcommands to additional hook events.** If lux
   gains a useful one-shot operation (cache warmup, diagnostics dump,
   workspace symbol prefetch), wiring it to a hook event requires editing
   `generate.go` and rebuilding the package.

4. **No per-event lifecycle control.** The choice of `PostToolUse` vs `Stop`
   vs `SessionEnd` for the formatter is hardcoded. Users may have legitimate
   reasons to prefer one over another (latency tolerance, batch size,
   interaction with subagents).

## Direction (sketched, not committed)

Introduce a new config file `~/.config/lux/claude-hooks.toml` (with
per-project override at `.lux/claude-hooks.toml`) that declares
*actions*: bindings of Claude hook events to lux subcommand invocations.

Lux's `generate-plugin` step always emits the full set of hook script
entries into Claude's `hooks.json` (so the hook surface is fixed at install
time). At runtime, each hook script invokes lux, which loads
`claude-hooks.toml` and decides whether to run, no-op, or branch on
filetype.

### Strawman schema

```toml
# ~/.config/lux/claude-hooks.toml

[[action]]
name    = "format-on-edit"
event   = "PostToolUse"          # PreToolUse | PostToolUse | Stop | SubagentStop | SessionStart | SessionEnd | UserPromptSubmit
matcher = "Edit|Write"           # Claude tool matcher; only meaningful for Pre/PostToolUse
command = "fmt"                  # MUST be a known lux subcommand
args    = ["{{file_path}}"]
disabled = false                 # default false
filetype_overrides = { markdown = { disabled = true } }

[[action]]
name    = "format-everything-on-stop"
event   = "Stop"
command = "fmt-all"
args    = []

[[action]]
name    = "deny-builtin-equivalents"
event   = "PreToolUse"
command = "hook"
args    = ["deny-builtins"]
```

**Field semantics (sketch):**

- `name` — required, unique. Merge key for per-project overrides.
- `event` — required, validated against the known Claude Code event set.
- `matcher` — optional. Forwarded verbatim to Claude's `hooks.json`. Ignored
  for events that don't take matchers.
- `command` — required. Validated at config-load time against lux's known
  subcommand set.
- `args` — optional. Templated with `{{var}}` substitution against the
  Claude hook stdin JSON (`{{file_path}}` is shorthand for
  `{{tool_input.file_path}}`).
- `disabled` — top-level kill switch.
- `filetype_overrides` — inline table keyed by LSP language ID. Each entry
  may override `disabled` and/or `args`.

### Open questions

1. **Templating syntax.** `{{file_path}}` (mustache-ish) vs `${file_path}`
   (shell-ish) vs `$CLAUDE_TOOL_INPUT_FILE_PATH` (env vars exported by the
   hook script wrapper, no templating in lux).

2. **Filetype override fields.** Just `disabled` and `args`, or also
   `command` (override the entire action)?

3. **Filetype identifier.** LSP `language_id` (matches `lsps.toml`),
   filetype name (matches `filetype/*.toml` filename), or formatter name
   (matches `formatters.toml`)?

4. **Default config shipped by lux.** Bake in code, ship a default
   `claude-hooks.toml` in the package, or load both?

5. **Buffer/state for multi-stage actions.** If we want to revisit a
   "PostToolUse buffer + Stop flush" pattern, where does the buffer live?
   Per-worktree state file? In-memory across hook invocations is
   impossible because each invocation is a fresh process.

6. **Dry-run / debug mode.** `lux hook --explain` to print what each event
   would do given the current config?

7. **Hook timeout configurability.** Today hardcoded at 60s in
   `generate.go`. Should `[[action]]` carry a `timeout` field?

8. **Custom commands beyond known lux subcommands.** Should the action
   `command` be limited to lux subcommands, or accept arbitrary shell
   commands (with the security implications that come with it)?

### Why this is *exploring* and not *proposed*

The v1 fix (`lux fmt-all` + Stop hook) intentionally bypasses this whole
config layer because:

- The tracer-bullet exercise on the bob worktree showed that the user's
  *actual* config has zero use cases for `exclude_filetypes`, `disabled`,
  or per-filetype action overrides — the markdown loop was already
  self-healed by commenting pandoc out, and no other filetype exhibits the
  loop in practice.
- A schema designed without concrete second and third use cases tends to
  ossify around the first one. `fmt-all` alone is the first; we need at
  least two more before locking the schema.
- The deny-builtins hook is generated by `go-mcp`, not by lux. Bringing it
  into a unified `claude-hooks.toml` requires either teaching `go-mcp` to
  read lux's config or splitting hook generation between go-mcp and lux at
  install time.

### Promotion to *proposed* requires

- v1 (`lux fmt-all` + Stop hook) shipped and used for 7+ days
- At least one concrete user request for a hook binding that v1 cannot
  express
- A second, distinct use case beyond fmt-all that the schema must support
  (candidates: PreToolUse customization, SessionStart cache warmup,
  PostToolUse linting on save, UserPromptSubmit context preloading)
- Resolution of open questions 1–3 above (templating syntax, filetype
  override fields, filetype identifier)

### Out of scope for this FDR

- Changing what `go-mcp` generates for `PreToolUse` deny-builtins
- The shape of the formatter routing config (`formatters.toml`,
  `filetype/*.toml`)
- LSP-driven formatting fallback (#87)
- Default generated-file exclusions for `fmt-all` (#88)
